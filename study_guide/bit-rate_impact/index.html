Tecnologías Multimedia- Study Guide - Milestone 7: Impact of the Bit-rate

Vicente González Ruiz - Depto Informática - UAL

October 20, 2020

1 Description

Along with the latency and its variation (jitter), another main aspect to consider about the transmission link used in an InterCom session is the transmission bit-rate (measured in bits per second or a multiple of this transmission capacity) that it can provide [1, 3]. This bit-rate depend on the maximum capacity (a characteristic closely related with the available bandwidth) and its congestion level (that basically depends on the load) of the link. In general, we can suppose that the capacity is constant along the time (the bandwidth provided by the link does not vary with the time). On the contrary, the congestion level is time-variying and quite unpredictable, because it depends on the behaviour of the network users.

In this milestone we are going to measure the impact of the transmission bit-rate on the QoE1 provided by the minimal implementation of InterCom. Similarly to the procedure used for measuring the impact of the latency and the jitter, we will use tc [2] to control the amount of data that an InterCom instance will be allowed to send in a local environment, with the aim of simulating a real running context. Notice that this upper bound in the bit-rate will also aﬀect to the loss of chunks because if the transmission bit-rate is smaller than the audio bit-rate, sooner of later the link will discard those chunks that can not be buﬀered in the retranmission nodes (routers and switches)2 .

2 What you have to do?

2.1 Estimate the bit-rate in an Internet link

Usually, we need to use a tool such as iPerf to measure the transmission bit-rate between two end-points in the Internet. This proceduce implies that we must have access to both hosts to install and run this program, privileges that are not always available.

Alternatively, we can estimate the link throughput using ping (see the previous milestone). At this point is important to realize that ping has been designed for measure latencies, not bit-rates, and that for this reason, we will be able to estimate throughputs rougtly.

The transmission bitrate available between two directly reachable IP devices can be estimated as

b = B tt , (1)

where B is the number of bits sent in a ping’s payload.

If we revisite the ping’s timeline and the Equation (2) of the previous milestone, we can see that

tt = RTT − 2tp 2 . (2)

Therefore, supposing that tp is constant between pings and using the average RTT, b can be considered as a good estimation of the near future.

So, for determining b we can:

  1. Run ping using the smallest possible payload that provides RTTs values.3 In this situation we can suppose that
    tp ≈ RTT 2 . (3)

    Notice that we should use the smaller RTT of a ping session of measurements for such approximation because the real link latency is a constant and the congestion of the link always increases it.

  2. Run ping using the largest possible payload to achieve that most part of the average RTT provided by ping corresponds to tt. This is necessary because the accuracy of the cronometer used by ping, in general, is not high enough to measure tt for small values of the payload size.
  3. Use the Equation 2 to ﬁnd tt as a function of the average RTT and the minimal tp.
  4. Finally, use the Equation 1 to determine b.

(Optional) In order to gain experience with all this stuﬀ, measure your bandwith with diﬀerent ping servers, try to compute the transmission bit-rate:

  1. In localhost.
  2. Between two hosts in your local network (your host and your router, for example). Notice that in this case, you should not obtain a bit-rate higher than the capacity of your network adapter.
  3. Between your host and a public host of the Internet. Try www.meter.net for example.

(Not optional) If the devices cannot ping each other, and supposing that there is only one point in the communication link that is ﬁltering the ICMP Echo Request traﬃc in each direction, then we can apply the same technique that was used in the previous milestone, which basically consists in compute the total RTT as the sum of the RTT from the devices that we want to connect (where the InterCom is it supposed to be run) to the ﬁlter (possiblely, one of your home routers).

2.2 Simulate the link

With the bit-rate vaule computed from the previous section, simulate such link locally with:

  sudo tc qdisc add dev lo root handle 1: tbf rate <bit-rate_in_kbps>kbit burst 32kbit limit 32kbit 
  sudo tc qdisc add dev lo root parent 1:1 handle 10: netem delay <average_delay_in_miliseconds>ms <maximum_average_deviation_in_miliseconds>ms <Pearson_correlation_coefficient_expressed_as_a_percentage>% distribution <uniform|normal|pareto|paretonormal>

Example:

  1. Check the current conﬁguration:
       tc qdisc show dev lo

    The output should be something like:

       qdisc noqueue 0: root refcnt 2
  2. Deﬁne the rule for the bit-rate control (example):
       sudo tc qdisc add dev lo root handle 1: tbf rate 200kbit burst 32kbit limit 32kbit
  3. Deﬁne the rule for the latency control (see previous milestone) (example):
       sudo tc qdisc add dev lo parent 1:1 handle 10: netem delay 100ms 10ms 25% distribution normal
  4. After adding these rules, this should be the conﬁguration:
       tc qdisc show dev lo

    The output should be something like:

       qdisc tbf 1: root refcnt 2 rate 200Kbit burst 4Kb lat 0us 
       qdisc netem 10: parent 1:1 limit 1000 delay 100ms  10ms 25%

Remember to delete the rules after their are unnecessary (example):

  sudo tc qdisc delete dev lo parent 1:1 handle 10: netem delay 100ms 10ms 25% distribution normal 
  sudo tc qdisc delete dev lo root handle 1: tbf rate 200kbit burst 32kbit limit 32kbit
